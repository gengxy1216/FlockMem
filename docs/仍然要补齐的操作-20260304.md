# MiniMem 下一步优化任务评估与完整落地方案（2026-03-04）

## 0. 评估基线（基于当前仓库）

当前已具备能力（作为本次方案输入约束）：

1. OpenClaw 插件已支持 `before_agent_start` 自动注入、`agent_end` 自动回写、`groupStrategy(shared/per_role/per_user)`、`memory slot` 接管（`integrations/openclaw-plugin/index.ts` + `install.ps1/sh`）。
2. MiniMem 当前配置来源是 `环境变量 + SQLite(app_config)`，并非单一 `config.json`（`src/evermemos_lite/config/settings.py`、`app_factory.py`、`api/routes/model_config.py`）。
3. 检索链路已存在多路融合与多种 rerank（RRF、启发式 rerank、时序 rerank），但没有独立 Cross-Encoder reranker。
4. 抽取链路已有 `rule/openai/chat_model` 三种 extractor，但“上下文压缩”主要还在 OpenClaw 插件侧做了截断式压缩（`compressMessages`），质量上限受限。
5. MiniMem 已有 MCP bridge（`integrations/minimem-mcp/server.py`），可被其他 Agent 体系接入。

结论：不是“从 0 到 1”，而是“从可用到可规模化与可治理”的改造。

## 1. 总体架构策略（先架构后编码）

采用三层目标架构，避免一次性重构风险：

1. 适配层（OpenClaw/其他 Agent）：负责身份映射、group 路由、技能入口适配。
2. 配置与策略层（MiniMem Core）：统一 `config.json`，并将检索/抽取/rerank 策略显式配置化。
3. 记忆处理层（写入-抽取-压缩-检索）：将“事实抽取”和“上下文压缩”解耦，形成可评估、可回退的双通道。

为何这个方案更简单、更一致：

1. 复用现有插件与 MCP，不推翻主链路。
2. 用“增量替换 + 兼容开关”处理配置统一，避免停机式迁移。
3. 所有新增能力默认可关闭，先保证稳定性再提升质量。

## 2. 逐条评估（价值/可行性/难度周期/影响）

评分说明：

- 价值：高/中/低（对目标“开箱可用 + 中枢化”的贡献）
- 可行性：高/中/低（结合现有代码基础）
- 难度：L/M/H
- 周期：按 1 名工程师的净开发日估算（不含需求等待）

### 1.1 读取 OpenClaw 配置并默认复用 primary 模型（可被 MiniMem config 覆盖）

现状：

1. 插件运行时没有文件系统权限（`openclaw.plugin.json` 中 `fileSystem=false`），不能直接在运行时读取 `~/.openclaw/openclaw.json`。
2. 安装脚本可以读写 OpenClaw 配置并写入插件 `entry.config`。
3. MiniMem 目前没有统一 `config.json`，模型配置是 env + SQLite runtime config。

价值：高  
可行性：中（需调整“读取时机”与配置链路）  
难度：M-H  
周期：4-6 天

建议改造：

1. 不在插件运行时读文件，改为“安装/同步脚本读取 OpenClaw 配置并写入插件 config”。
2. 在插件 config 新增：`inheritPrimaryModel=true`、`primaryModelSnapshot{provider,baseUrl,model}`。
3. MiniMem 侧新增“来源优先级”：`explicit config.json > OpenClaw primary snapshot > env default`。
4. 增加一次性同步命令：`minimem-openclaw-sync`（用于 OpenClaw primary 改动后的重同步）。

影响评估：

1. 稳定性：中风险，集中在配置优先级冲突；可通过显式优先级与日志消除歧义。
2. 安全性：中风险，模型密钥透传需脱敏与最小暴露；日志禁止打印 API key。
3. 兼容性：中风险，需兼容旧插件配置；通过默认 `inheritPrimaryModel=false` 可平滑升级。

结论：应做，优先级 P0。

### 1.2 读取 Agents/channels，生成 sender-id 与 share/group 路由

现状：

1. 插件已支持 `autoSenderFromAgent` 与 `groupStrategy`，但还没有“按 OpenClaw Agents/channels 的显式映射表”。
2. 目前 sender/group 路由偏通用，尚未形成多 Agent 中枢治理规则。

价值：高  
可行性：高  
难度：M  
周期：5-7 天

建议改造：

1. 在安装/同步阶段解析 OpenClaw 的 `agents/channels`，生成并写入插件 config：
   - `senderMap: {agentId -> senderId}`
   - `channelGroupMap: {channel -> groupId}`
   - `sharePolicy: {group -> readableAgents/writableAgents}`
2. 插件 `resolveScope` 先查映射，再回退到 `groupStrategy` 默认策略。
3. 记忆 metadata 写入 `agent_id/channel/task_id`，便于后续过滤与审计。

影响评估：

1. 稳定性：低到中风险，路由规则增加后需防止“串组”；需加路由回退与告警。
2. 安全性：中风险，跨 Agent 共享会扩大读取面；需加可选 ACL（最少先做读写白名单）。
3. 兼容性：低风险，未配置映射时沿用当前行为。

结论：应做，优先级 P0。

### 1.3 接管 OpenClaw memory slot + 明确压缩策略

现状：

1. 安装脚本已把 `plugins.slots.memory = minimem-memory`，已有长期记忆接管基础。
2. 压缩策略当前是“最后若干消息拼接+截断”，语义压缩能力不足。

价值：高  
可行性：高  
难度：M  
周期：3-5 天

建议改造：

1. 保留现有 slot 接管机制，补“可观测压缩策略”：
   - `compression_mode`: `truncate | llm_summary | hybrid`
   - `compression_budget_tokens/chars`
   - `compression_min_turns`
2. 默认 `hybrid`：先规则压缩，再在预算内做摘要；失败自动降级 `truncate`。
3. 压缩结果 metadata 补齐：`source_turns`、`compression_ratio`、`method`、`error_fallback`。

影响评估：

1. 稳定性：低风险（有回退策略）。
2. 安全性：低到中风险（摘要会处理原文，注意敏感信息脱敏配置）。
3. 兼容性：低风险（保持现有默认行为可兼容）。

结论：应做，优先级 P0。

### 1.4 复用 OpenClaw skills（含 find-skills 自动安装与文档摘要）

现状：

1. 插件本身不是 skill 执行器，且无文件系统权限，不适合直接执行文件解析。
2. 该需求本质是“文档摄入流水线”而非单点插件逻辑。

价值：中-高  
可行性：中  
难度：H  
周期：7-10 天

建议改造：

1. 明确职责边界：skill 执行在 Agent/OpenClaw 侧，MiniMem 接收“解析后的结构化文本”。
2. 增加 `ingest` 接口契约（MCP 或 HTTP）：`source_type/source_uri/raw_text/summary/chunks/metadata`。
3. 提供 bootstrap 脚本做“可选自动安装 find-skills”，但默认不强制联网安装（企业环境合规要求）。
4. 摘要长度采用预算策略：`short/medium/long` 或 `token_budget`，由文档长度驱动。

影响评估：

1. 稳定性：中风险（外部 skill 质量波动）。
2. 安全性：高风险点（外部命令执行、供应链、远程仓库安装）；必须白名单 + 可审计日志。
3. 兼容性：中风险（不同技能生态接口不统一）；需定义标准 ingest schema。

结论：应做但后置，优先级 P1。

### 2.1 统一 `config.json`，实现配置文件化

现状：

1. 当前是 env + SQLite `app_config` 双轨，实际存在“来源分叉”。
2. UI/API 修改模型配置时会写入 SQLite。

价值：高  
可行性：中  
难度：H  
周期：8-12 天

建议改造：

1. 引入 `config.json` 作为主配置源，结构分区：
   - `server`、`models`、`retrieval`、`extractor`、`storage`、`security`、`integration`
2. 迁移策略：
   - 启动时执行一次 merge：`config.json <- (SQLite + env)`，并写迁移标记。
   - 过渡期保留读取兼容，写入只写 `config.json`（避免新旧并存）。
3. `model-config` API 改为操作 `config.json`（必要字段加热加载）。

影响评估：

1. 稳定性：中-高风险（配置路径变更影响启动链路）。
2. 安全性：中风险（密钥落盘）；需支持 `config.secrets.json` 或环境变量覆盖敏感项。
3. 兼容性：中风险；需提供迁移脚本和旧字段映射表。

结论：必须做，优先级 P0。

### 2.2 配置支持本地向量模型和排序模型（参考 QMD 思路）

现状：

1. embedding provider 目前仅 openai-compatible 路径。
2. 没有独立 reranker provider 抽象。

价值：高  
可行性：中  
难度：H  
周期：6-9 天

建议改造：

1. 抽象 `EmbeddingProviderFactory`：`openai_compatible | local_sentence_transformers`。
2. 新增 `RerankProvider`：`none | local_cross_encoder | openai_compatible_rerank`。

影响评估：

1. 稳定性：中风险（本地模型资源占用显著）。
2. 安全性：低到中风险（本地模型通常更可控，但仍需模型来源可信校验）。
3. 兼容性：中风险（依赖包与平台差异，Windows/Linux CUDA 适配）。

结论：应做，优先级 P1（紧随 2.1）。

### 2.3 检索链路是否加入 rerank（专项评估）

结论：需要，但应“按条件启用”，不是全量强制。

依据：

1. 现有链路已有基础 rerank，但偏启发式，面对长文档/近义冲突仍会误排。
2. Cross-Encoder rerank 对 Top-K 精度提升明显，但增加时延。

价值：高  
可行性：高  
难度：M  
周期：3-5 天（含 A/B 评测脚本 2 天）

建议策略：

1. 默认规则：当 `hybrid/agentic` 且召回候选 > `rerank_trigger_k` 时触发 rerank。
2. 预算控制：只 rerank 前 `N` 个候选（如 20/30），并设置超时降级。
3. 输出可观测字段：`rerank_applied/model/latency_ms/delta_score`。

影响评估：

1. 稳定性：中风险（耗时上升）；通过超时 + fallback 控制。
2. 安全性：低风险（主要是服务可用性风险）。
3. 兼容性：低风险（可配置关闭，兼容原链路）。

结论：应做，优先级 P0.5（介于 P0 与 P1）。

现状：

1. 已有 OpenClaw plugin 与独立 MCP server，接入方式分散。
2. 缺少统一接入规范（身份、group、权限、审计字段）。

价值：中-高

影响评估：

1. 稳定性：低风险（主要是文档与适配层工作）。
2. 安全性：中风险（多入口需要统一认证策略）。
3. 兼容性：低风险（新增规范，不破坏旧接口）。

结论：应做，优先级 P1。

### 2.5 抽取链路复杂且压缩上下文效果不足，需加强

现状：

1. 当前抽取兼顾事实提取、重要度估计、画像/前瞻识别，功能不少但目标混杂。
2. 上下文压缩主要在插件端做拼接截断，未形成“语义摘要优先”的稳定链路。

价值：高  
可行性：中  
难度：H  
周期：7-12 天

建议改造：

1. 拆分双通道：
   - `FactExtractor`：面向结构化记忆入库（事实/画像/计划）
   - `ContextCompressor`：面向 prompt 注入（任务相关摘要）
2. 压缩质量策略：
   - 约束输出 schema（topic/key_points/open_questions/actions）
   - 基于 query 的动态摘要粒度
3. 引入“压缩有效性评估”：
   - 压缩前后 token 比
   - 引用命中率
   - 回答正确性回归集

影响评估：

1. 稳定性：中风险（链路拆分会触发较多回归面）。
2. 安全性：低到中风险（摘要层需敏感信息处理策略）。
3. 兼容性：中风险（新旧字段与接口输出需对齐）。

结论：必须做，优先级 P0（但安排在配置统一后实施）。

### 2.6 MiniMem 自身支持 skills（markitdown/ppt 等）

现状：

1. MiniMem 当前核心是记忆服务，不是技能运行平台。
2. 但已有 MCP 基础，适合作为“技能结果接入层”。

价值：高  
可行性：中  
难度：H  
周期：8-12 天

建议改造：

1. 不直接内置所有 skill，采用“Skill Adapter”模式：
   - `skill_runner`（外部）
   - `ingest_adapter`（MiniMem 内）
2. 支持标准化输入：
   - `content_type`：`text/markdown/pdf/pptx/docx`
   - `parsed_text`、`summary`、`attachments_meta`
3. 首批白名单 skill：`markitdown`、`pptx parser`、`pdf extractor`。

影响评估：

1. 稳定性：中风险（外部 skill 波动）。
2. 安全性：高风险（命令执行与供应链）；必须做签名校验/来源白名单/沙箱。
3. 兼容性：中风险（不同 skill 输出格式差异大）。

结论：应做但后置，优先级 P1-P2。

## 3. 综合优先级与实施排期（重排版）

重排原则：

1. 先解决配置源分叉，再做链路增强与生态扩展。
2. 任何依赖配置项的新能力，均在 `2.1` 之后上线。

## Phase 0（第 1-2 周，配置统一先行）

1. 2.1 `config.json` 主配置落地 + 迁移脚本 + 回滚脚本。
2. 1.1 primary 模型继承能力（安装期同步 + 优先级覆盖）并入新配置体系。
3. 2.3 rerank 开关先配置化（先不引入新 reranker 模型）。

交付标准：

1. 启动后配置来源唯一可解释（`config.json` 主源）。
2. 历史部署可无损迁移，可一键回滚。
3. 配置改动对运行链路可热加载或可预测重启生效。
4. 建立首版性能基线（P50/P95）并写入发布门禁。

## Phase 1（第 3-4 周，OpenClaw 中枢能力稳定化）

1. 1.2 sender/group 映射最小版（映射表 + 回退策略 + metadata）。
2. 1.3 memory slot 接管后的压缩策略治理（`truncate/llm_summary/hybrid`）。

交付标准：

1. OpenClaw 多 Agent 的隔离/共享路由可控且可审计。
2. 压缩链路失败可自动降级，主流程不中断。
3. `before_agent_start` 注入链路 P95 不高于基线 + 15%。

## Phase 2（第 5-6 周，检索与抽取质量提升）

1. 2.5 抽取/压缩双通道重构（`FactExtractor + ContextCompressor`）。
2. 2.2 本地 embedding/rerank provider 抽象。

交付标准：

1. 压缩比、引用命中率和回答一致性有可测提升。
2. 本地模型可切换、可降级，服务时延在预算内。

## Phase 3（第 7-8 周，多 Agent 与技能生态接入）

1. 2.4 统一其他 Agent 接入规范（slot/MCP/HTTP 三模式）。
2. 1.4 + 2.6 skill 适配与 ingest 标准化。

交付标准：

1. 至少 2 类外部 agent 与 2 类文档 skill 稳定接入。
2. 安全基线（白名单/审计/供应链约束）通过。

## 4. 验证与测试方案（核心路径三类验证）

每个 Phase 至少覆盖：

1. 正常路径：标准写入/检索/注入流程。
2. 关键边界：空配置、错误 provider、大文档、跨 group 读取。
3. 错误恢复：模型超时、rerank 失败、skill 执行失败后的降级链路。

新增测试建议：

1. `tests/test_openclaw_scope_mapping.py`
2. `tests/test_config_json_migration.py`
3. `tests/test_rerank_gate_and_fallback.py`
4. `tests/test_context_compressor_quality.py`
5. `tests/test_ingest_skill_adapter_schema.py`

## 5. 风险清单与缓解

高风险：

1. 配置统一导致历史部署启动失败。  
缓解：双读单写过渡 + 一键回滚到旧读取链路。
2. 外部 skill 供应链与命令执行风险。  
缓解：白名单仓库、版本锁定、禁用默认自动安装、审计日志。
3. rerank/本地模型拉高延迟。  
缓解：阈值触发 + 超时降级 + 运行时开关。

## 5.1 性能预算与硬约束（新增，必须执行）

总体原则：

1. 所有新增能力默认 `off` 或“条件触发”，不得默认走最重链路。
2. 任一增强组件超时后必须 fail-open，回退到基础检索，不阻断主流程。
3. 发布必须通过延迟门禁，不满足预算禁止合入主分支发布。

延迟预算（建议首版）：

1. `GET /api/v1/memories/search`：
   - P50 <= 350ms
   - P95 <= 1200ms
2. OpenClaw `before_agent_start` memory 注入：
   - P50 <= 450ms
   - P95 <= 1500ms
3. `POST /api/v1/memories`（不含外部大模型抽取）：
   - P50 <= 220ms
   - P95 <= 700ms

组件预算与降级：

1. rerank：
   - 仅在候选数 > `rerank_trigger_k` 时触发
   - `rerank_top_n` 默认 20
   - `rerank_timeout_ms` 默认 220ms，超时立即回退原排序
2. query rewrite / verifier：
   - 总预算 `phase23_budget_ms` 默认 300ms
   - 超预算直接跳过二轮
3. compressor：
   - `compression_timeout_ms` 默认 280ms
   - 失败回退 `truncate`

资源预算（单机）：

1. 常驻内存增长上限：相对基线 <= +20%。
2. CPU 占用上限：峰值不超过 1.6x 基线（10 分钟滑窗）。
3. 本地模型启用时需支持并发阈值与队列长度限制，超过阈值自动旁路。

观测与门禁：

1. 每次请求输出：`trace_id`, `phase_latency`, `fallback_reason`, `rerank_applied`。
2. CI 增加性能冒烟：固定数据集对比基线，P95 回归 > 20% 直接失败。
3. 生产灰度阶段按 5% -> 20% -> 50% 放量，任一阶段超预算自动回滚。

## 6. 本轮结论（用于决策）

必须优先执行（P0，按顺序）：

## 7. 额外优化建议（新增）

### 7.1 可观测性与回归护栏（建议 P0.5）

1. 增加检索全链路 trace 字段：`trace_id/phase/latency_ms/fallback_reason`。
2. 固化 nightly 回归集：至少覆盖身份问答、时间线问答、跨 Agent 协作问答。
3. 建立发布门禁：关键指标退化超过阈值自动阻断发布（含性能 P95）。

价值：高；难度：M；周期：3-4 天。

### 7.2 幂等与写入治理（建议 P1）

1. 写入接口增加幂等键（`idempotency_key`），避免插件重试导致重复记忆。
2. 增加 TTL/归档策略，避免长期运行后噪声膨胀。

价值：中-高；难度：M；周期：2-3 天。

### 7.3 安全基线加强（建议 P1）

1. API 鉴权从“可选”升级为“生产默认开启”。
2. 技能接入统一白名单和版本锁，禁止未审计源自动安装。
3. 关键配置与密钥脱敏输出，审计日志最少留痕 30 天。

价值：高；难度：M；周期：3-5 天。
